引入

​ Git是目前世界上最先进的分布式版本控制系统

​ 不但能自动记录每次文件的改动，还可以让同事协作编辑。这样就不用自己管理一堆类似的文件了，也不需要把文件传来传去：这就是版本控制。

CVS和SVN，这些集中式的版本控制系统不但速度慢，而且必须联网才能使用。

但是分布式版本控制系统通常也有一台充当“中央服务器”的电脑，但这个服务器的作用仅仅是用来方便“交换”大家的修改，没有它大家也一样干活，只是交换修改不方便而已。

  • 可以看出，git的世界里面，每一台机器都有自己的版本库，都可以有一个标识：

  • 版本库

    版本库又名仓库，英文名repository，你可以简单理解成一个目录，这个目录里面的所有文件都可以被Git管理起来，每个文件的修改、删除，Git都能跟踪，以便任何时刻都可以追踪历史，或者在将来某个时刻可以“还原”。

    • 创建一个版本库

      当前目录还会有一个隐藏文件夹.git（可以用ls -ah查看），里面有Git跟踪管理版本库的信息。

    • 添加文件到版本库

      首先这里再明确一下，所有的版本控制系统，其实只能跟踪文本文件的改动，比如TXT文件，网页，所有的程序代码等等，Git也不例外。版本控制系统可以告诉你每次的改动，比如在第5行加了一个单词“Linux”，在第8行删了一个单词“Windows”。而图片、视频这些二进制文件，虽然也能由版本控制系统管理，但没法跟踪文件的变化，只能把二进制文件每次改动串起来，也就是只知道图片从100KB改成了120KB，但到底改了啥，版本控制系统不知道，也没法知道。

查看仓库状态

读取存档

​ 我们不断对仓库作出修改，这就产生一个个版本/存档，当我们出错了的时候，git允许我们回退到之前的版本。

  • 查看版本

    git log

    注：版本号是SHA1计算出来的；HEAD标识的是最新版本；--pretty=oneline可以略去作者和时间信息

  • 选取版本回退

    这时候就看出，commit时候加的说明的重要性了。

  • 想穿回来

    回去以后，log一看，后面的不见了。如果还记得版本号的话（*—*）

    其实就是HEAD指针在移动吧。

工作区 暂存区 版本库

​ 工作区就是git init在的那个目录。

​ 这个区里面有一个隐藏目录.git，但不算工作区，是版本库。

​ 隐藏目录里面有一个index文件（暂存区）。

​ 所谓add，就是将文件放到暂存区；所谓commit，就是将暂存区的所有文件都转移到当前分支.

  • 撤销修改
    • 还没add：手动改回去；git checkout -- file 或者 git restore file恢复到最近一次add或者commit的状态
    • 已经add：git restore --staged file 或者 git reset HEAD file这样暂存区就干净了，就退回到了上一种情况。
    • 如果已经提交到本地的库了，还没上传，就是版本回退了。git reset commit id or HEAD~?

删除文件

​ 直接rm删除了工作区里面的文件，版本库的文件还在。

  • 如果真的要删：再删除暂存区的git rm file。然后提交到版本库git commit -m ""
  • 如果不要删除：那就是将暂存区的导回来 git checkout -- file or git restore file

远程仓库

​ 在之前的章节里面，git仓库都是部署在本地电脑的。现在，我们尝试跨机器。实际上，我们是找一台机器当作服务器，一直开机，人们可以pull&clone。github这个网站就是提供git仓库托管服务的，可以获得免费的远程仓库。

​ 远程库和本地库的一些操作

  • 本地仓库和远程仓库之间的连接通过SSH加密：

    • 创建密匙ssh-keygen -t rsa -C "邮箱" 创建完成，在.ssh文件夹里面就会有公匙和私匙
    • 在github个人设置里面添加SSH KEY

  • 关联远程仓库 推送内容

    分布式版本系统的最大好处之一是在本地工作完全不需要考虑远程库的存在，也就是有没有联网都可以正常工作，而SVN在没有联网的时候是拒绝干活的！当有网络的时候，再把本地提交推送一下就完成了同步

  • 从远程仓库克隆

    上一小节是从本地库到远程库。现在是由远程库到本地库。

分支管理

​ 并行开发，互不干扰，可以合并。

创建分支以及一些其他操作

​ 对于每一次提交（commit），git将其归档，这一条时间线就是一个分支。一般我们的最开始那个分支叫做main（原来是master hh），版本库里面HEAD指向的其实是分支，也就是HEAD->master->commit node。

​ 每次提交，master分支向前伸展，线也就变长了。一旦创建一个新的分支dev，那么这个分支（指针）就会指向当前的提交点，再由HEAD指向dev，表明当前分支是dev。然后就让dev

分支合并冲突

Fast-forward，换句话说，如果顺着一个分支走下去可以到达另一个分支的话，那么 Git 在合并两者时，只会简单地把指针右移，因为这种单线的历史分支不存在任何需要解决的分歧，所以这种合并过程可以称为快进（Fast forward）

​ 现在我们考虑这样一种情况：从main分支switch -c一个分支，然后我们对一个文本文件修改一次，提交到版本库；我们再切回来，又对这个文件修改，提交，然后合并分支。这样，这个文本文件就出问题了，该选哪个呢？

​ 这就需要我们手动做一些调整。也就是对冲突的文本进行编辑，然后提交到版本库。可以认为合并冲突的时候，不冲突的部分已经合并好了。

git log --graph可以查看合并情况

合并分支的一些策略

保留合并历史

合并分支时，加上--no-ff参数就可以用普通模式合并，合并后的历史有分支，能看出来曾经做过合并，而fast forward合并就看不出来曾经做过合并。

bug

​ git的分支，迅捷有力，有bug，直接开一个branch来修复。

  • stash

    假如我们要切到另外一个分支去解决bug，但是当前分支工作区文件并没有改好，我们先使用git stash来保存，使得工作区clean。

  • （必须的）我们在需要修复bug的分支checkout -b一个新的分支，然后解决，再返回merge

  • 切回工作分支，恢复工作区内容。

    git stash apply && fit stash drop

    or

    git stash pop

    git stash list可以看stash内容；可以git stash apply@{?}来选取对哪一个stash操作。

  • cherry-pick

    git cherry-pick commit_id可以方便地将修复bug部分的提交施加在当前分支上。

功能分支

​ 每添加一个新的功能，建议新增一个分支feature

​ 如果功能弃用，删除之，需要注意：没merge的分支需要强制删除git branch -D ?

标签

  • 切换到需要打标签的分支，然后：git tag "tag_name" (commit id 默认最新的)
    • -a tag_name
    • -m 注释
  • git tag查看该分支所有标签

  • git show tag_name查看该标签信息

  • 删除

    git tag -d tag_name

  • 推送到远程仓库

    git push origin tag_name

    特别的：

    ​ git push origin --tags推送所有标签

    远程删除：

    ​ 先本地删除

    ​ 再git push origin :refs/tags/tag_name

关于git存储原理浅究

​ 如前文所述，本地库根目录下有一个隐藏的文件夹.git，里面存储了版本库信息。

git里面的objects:

假设完成一次add&&commit：

  • blob

    当我们add一个工作区的文件到暂存库，.git/objects目录里面就会产生一个该类型的对象。

    它存储一个文件的内容，别无长物。就这些内容会被SHA1成一个哈希值，作为该对象在这个仓库里面的一个身份牌（这也是在该objects下的对应目录名字+目录内文件名字）

  • tree

    当我们commit一次，在.git/objects里面又会产生一个tree类型的对象 。

    它存储当前的目录结构信息（shot）：包含这一次commit时候每一个文件的权限、类型、对应的身份证（SHA1值）、以及文件名。

    它也会有自己的身份牌（SHA1出来的哈希值）

  • commit

    当我们commit一次，除了tree以外，我们还会找到一个新的类型的对象。

    它存储一次提交信息，包括对应tree的哈希值，上一个提交的哈希值（这里由于是第一个提交，所以没有父节点。在一个merge提交中还会出现多个父节点），提交的作者以及提交的具体时间，最后是该提交的信息。

    当然，commit也有自己的SHA1值作为身份牌。

git里面的分支存储

​ 我们都知道，git里面的分支操作是很快的，为什么呢？因为分支本身只是一个“类指针”，不同于C里面存储逻辑地址的指针，这里的指针存储的是commit对象的SHA1值。所以所谓分支只是指向不同提交节点的“指针”。

这样，结合git的三个区，较容易理解：当我们修改工作区的文件时候，.git里面没有变化；当我们add时候，一方面增加了相应的blob对象，一方面改变了.git/index（或许是所谓暂存区）的内容；当我们commit的时候，我们增加了相应的tree和commit对象，并且将当前分支和HEAD都改变指向到最近的commit对象。而创建新的分支也不过是在.git/refs里面增加了新的文件。
