搜索引擎是怎么工作的呢？

肯定是要做search，那么怎么查找呢？倒序文件索引。

它的基本结构如下：

term dictionary and posting list，好像一个字典。

操作

创建一个索引

在创建时候，我们需要使用spider来爬取所有的document。

  • key

    • 怎么知道一个term是一个term呢？

      token analyzer

      • word stemming（词干提取）

        etc. in English

      • stop words(stop filter)

        去除掉一些没有意义的词

    • 怎么从一张很大的dictionary里面找到我们的term呢？

      • 搜索树

      • 哈希 常数时间，但是很难实现，而且范围查找很耗时间，因为结构类似的词可能隔得很远。

    • 将数据保存到外存里面。数据太多了，动态保存。

策略

Distributed indexing

​ 有很多台主机。将它们看作图上的结点，每一个结点存储a subset of collection。

  • term-partitioned index

    读入一个term，选取适当主机（每台主机的term的首字母具有共性），插入存储

  • document-partitioned index

    每台主机存储一定量的document

Dynamic indexing

​ 当DOC进来的时候，如果term在dictionary了，我们就做posting list update，不然得插入新的结点，这会很耗时间。

Compression

一个是压缩dictionary，一个是压缩posting list里面指向DOC的“指针”。

但是，后者时候会带来寻找的困难呢（要做求和

Thresholding

  1. 没必要返回所有的匹配页面。我们设定一个阈值，把那些重要的，相关度高的结果返回。如果多关键字搜索，做intersection前可能遗漏了一些
  2. 怎么判断重要与否呢？或许可以看它的frequency，这个值越小，意味着越少的DOC能包含它，我们做一个递增的排序，按照前面的关键字搜索，一步步扩大到结果差不多为止，不必全部搜索求交集。

衡量标准

对于一个搜索引擎，我们需要关注

  • 它做索引有多快。

    number of docs per hour

  • 搜索有多快

    latency as a function of index size(不能直接用latency， 因为不能保证index是一样大的)

  • 查询语句的支持

    比如 关键字1-关键字2，那么查询的结果就应该包含前者但是不含后者。

  • 用户体验感？

    • 数据，是否都呈现了含关键字的结果
    • 信息，相关度怎么样
      • benchmark document collection
      • benchmark suite of queries
      • binary assessment
        • page rank
        • semantic web
